# @autonoma/ui

The Autonoma AI frontend - a single-page application for creating, managing, and reviewing agentic end-to-end tests across web and mobile platforms.

## Tech Stack

- **React 19** with React Compiler (via `babel-plugin-react-compiler`)
- **Vite 6** - build tool and dev server (port 3000)
- **TanStack Router** - file-based routing with auto-generated route tree
- **TanStack Query + tRPC** - type-safe data fetching against the API server
- **Tailwind CSS v4** - styling (via `@tailwindcss/vite` plugin)
- **@autonoma/blacklight** - shared component library (Radix UI + Tailwind + CVA)
- **@autonoma/pond-ui** - additional shared UI components
- **better-auth** - authentication client (session, org switching, API keys)
- **Zustand** - lightweight client state management
- **Sentry** - error tracking and breadcrumbs
- **PostHog** - product analytics and user identification

## Getting Started

From the monorepo root:

```bash
pnpm install
pnpm db:generate    # required for tRPC type inference
pnpm dev            # starts both UI (port 3000) and API (port 4000)
```

Or run the UI alone:

```bash
cd apps/ui
pnpm dev            # starts Vite dev server on port 3000
```

The dev server proxies `/v1` requests to `http://localhost:4000` (the API), so you need the API running for the app to function.

### Build

```bash
pnpm build          # runs typecheck then vite build
pnpm preview        # serves the built dist/ locally
```

## Environment Variables

Defined in `src/env.ts` using `@t3-oss/env-core` with Zod validation. All variables use the `VITE_` prefix.

| Variable | Required | Default | Description |
|---|---|---|---|
| `VITE_API_URL` | No | `http://localhost:4000` | API server URL |
| `VITE_INTERNAL_DOMAIN` | No | `autonoma.app` | Internal domain for auth base URL resolution |
| `VITE_TEMPORAL_URL` | No | - | Temporal UI dashboard URL |
| `VITE_SENTRY_DSN` | No | - | Sentry DSN for error tracking (disabled if unset) |
| `VITE_POSTHOG_KEY` | No | - | PostHog project API key (analytics disabled if unset) |

## Architecture Notes

- **File-based routing** - TanStack Router plugin scans `src/routes/` and auto-generates `routeTree.gen.ts`. Add a new file to add a route.
- **Type-safe API layer** - tRPC types flow from `@autonoma/api` to the frontend. Never manually define API response types. Use `trpc.<router>.<procedure>.queryOptions()` with `useSuspenseQuery` for reads.
- **Route context** - The router context provides `auth`, `queryClient`, and `trpc` to all routes, set up in `main.tsx`.
- **Onboarding URL model** - onboarding uses a single canonical route at `/onboarding` with `step` search state (for example `/onboarding?step=working`). Legacy onboarding subpaths redirect to the canonical URL.
- **Proxy in dev** - Vite proxies `/v1`, `/ingest`, and `/.well-known/oauth-*` to the API server, so the frontend and API share the same origin during development. The `/.well-known/oauth-*` proxy mirrors the production `nginx.conf.template` rule: Better Auth advertises the MCP OAuth discovery metadata at the app origin, but the API serves it - without the proxy those requests fall through to the SPA and the MCP OAuth handshake never starts.
- **Static SPA** - No SSR. Builds to static files in `dist/`, served by nginx in production.
- **Code splitting** - TanStack Router `autoCodeSplitting` is enabled in `vite.config.ts`, so each route's component and loader ship as their own lazily-loaded chunk (heavy libs like `recharts`, `react-markdown`, and `prism-react-renderer` load only with the routes that use them). A `manualChunks` function isolates the framework core loaded on every page - react, router/query, base-ui/radix, Sentry, PostHog - into stable named vendor chunks so an app-code deploy does not bust their browser cache.
- **Cache headers** - `nginx.conf.template` caches content-hashed `/assets/*` as `immutable` (one year), but serves `index.html` and the PWA `sw.js` with `Cache-Control: no-cache` so they always revalidate. The SPA entry keeps a stable URL and cannot be content-hashed, so without this a client can keep loading an old shell that points at asset hashes the latest deploy already deleted. In production the app sits behind CloudFront: these origin headers only take effect if the CloudFront cache behavior honors origin `Cache-Control` (and a deploy should invalidate `/` and `/index.html` at the edge).
- **PWA** - `vite-plugin-pwa` (`registerType: "autoUpdate"`) precaches build assets for offline use; the `no-cache` on `sw.js` is what lets clients pick up a new service worker on the next visit.
- **tRPC batching** - Queries are batched via `httpBatchLink`, except `FormData` uploads which use `httpLink`.
- **Deferred support chat** - PostHog analytics initializes with the app, while the conversations widget and its external bundle load after the browser `load` event.
- **ESM only** - No CommonJS. All imports use bare specifiers without `.js` extensions.
