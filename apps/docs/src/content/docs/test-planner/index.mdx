<p class="lead">The planner is a command-line tool that reads your codebase and generates a complete end-to-end test suite: the pages and flows to cover, the data each test needs, and the helpers that create and clean up that data. One command, and it runs on managed Autonoma credits - no LLM API key required.</p>

![The planner reads your codebase and produces a knowledge base, scenarios, and a test suite](/img/test-planner/pipeline.jpg)

## Run it

You launch the planner from the **Finish setup → Upload test artifacts** step in the dashboard, which hands you a ready-to-paste command with your token and generation id baked in:

```bash
AUTONOMA_API_TOKEN=... AUTONOMA_GENERATION_ID=... npx @autonoma-ai/planner@latest
```

That `npx` line downloads and runs the planner, published on npm as `@autonoma-ai/planner` (the installed command is `autonoma-planner`). Run it from the root of the repo you want to test. It works against your **frontend** (to map pages and flows) and your **backend** (to map data models and wire up test data). If those live in separate repos, run it where it can reach both.

It works through seven steps; everything it produces is readable live in the dashboard as it's written.

## Watch it work

A full run can take an hour or more, so the pipeline phase runs inside a live dashboard right in your terminal: the steps and their progress on the left, every generated file in the middle, and the document being written right now on the right - streaming live from disk so you can read the knowledge base, scenarios, and tests as they're produced. An activity feed at the bottom shows exactly what the agent is doing at any moment, and the top bar tracks elapsed time and a single estimated time remaining.

Navigate with the arrow keys (or `h/j/k/l`): move between panels, open any produced file, and scroll it while the run continues. Press `f` to jump back to following the newest file. Progress is saved continuously - Ctrl+C twice exits, and `--resume` picks up where you left off.

## What it does

| # | Step | What it does | Output |
| --- | --- | --- | --- |
| 1 | **Map your project structure** | Finds your frontend(s) and backend(s) so later steps scan only what matters. | `project-map.json` |
| 2 | **Find your pages** | Maps every page and route in your app. | `pages.json` |
| 3 | **Build a knowledge base** | Learns your features, flows, and UI patterns. | `AUTONOMA.md` |
| 4 | **Map your data models** | Finds what your app stores and how each record is created. | `entity-audit.md` |
| 5 | **Design test scenarios** | Decides the realistic data each test runs against. | `scenarios.md` |
| 6 | **Set up test data** | Hands the [Environment Factory](/environment-factory/) integration to your own coding agent, which implements it, validates it live, and produces the recipe. | `recipe.json` |
| 7 | **Generate the tests** | Writes the E2E tests as natural-language markdown, covering every page and feature. | `qa-tests/` |

### Step 6: your coding agent wires the test data

Setting up test data means wiring the [Environment Factory](/environment-factory/) into your backend - real code in your repo. Instead of handing you instructions to paste somewhere, the planner launches your **locally installed Claude** in one session, the same way `git commit` opens your editor: the dashboard steps aside, your agent takes the terminal, and control returns when it's done.

The agent installs the SDK, builds the endpoint, registers factories that call your app's real creation code, generates the test-data recipe, and validates every entity against your locally running app - creating records, checking your database, tearing them down, and checking they're gone. When it reports the integration complete, the planner uploads the recipe and moves on to test generation.

By default the session is fully autonomous; pass `--permission-mode` to make it ask before edits or commands instead. If no supported agent is installed (or you decline), the planner writes the full instructions to `~/.autonoma/<app>/integration-prompt.md` so you can implement them with any assistant, then continue with `--resume`.

## Reviewing the output

Everything the planner produces streams into the dashboard's file list the moment it's written - open any file and read it while the run continues. If a step fails, the planner stops and asks whether to retry, retry with a steering note, or stop; progress is saved either way.

## Commands and flags

```bash
autonoma-planner            # run the pipeline (default command)
autonoma-planner status     # show progress for the current project
autonoma-planner upload     # re-upload already-generated artifacts after a failed upload

# Useful flags
--project <path>            # target a repo other than the current directory
--frontend <path>           # in a monorepo, the frontend directory to plan tests for
--backends <path,path>      # the backend/data-layer directories it depends on
--step <name>               # run or re-run a single step
--resume                    # continue from where a previous run stopped
--non-interactive           # run unattended, plain output (CI)
--model <id>                # pick a different Autonoma-hosted model (still no key needed)
--agent <name>              # which coding agent handles the test-data step (default: auto-detect)
--permission-mode <mode>    # its autonomy: bypassPermissions (default) | acceptEdits | default
```

## Output

Artifacts are written to `~/.autonoma/<project-slug>/` as they're produced. When you run the planner from the dashboard's Finish setup flow, the knowledge base, scenarios, recipe, and generated tests are uploaded to Autonoma automatically at the end - ready to run against your preview environments.

:::note
The planner and the [Environment Factory](/environment-factory/) work together: the planner **designs** the scenarios and generates the tests; the Environment Factory **provisions** the real, isolated data those tests run against, on every run.
:::
